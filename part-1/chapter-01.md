# 第 1 章 FDE 的真实工作流

很多人接到 FDE 工作，第一周最大的迷茫不是"技术不会"，是不知道时间花在哪儿是对的。

跟客户开会算不算干活？写 demo 算不算交付？花一上午标 50 条评估集算不算正事？跟客户运维拉群对 SSO 配置算不算 FDE 的范围？

这一章要回答的是这个问题。它不教具体技术，目的是在你打开 IDE 之前，先建起一个"我此刻应该在做什么"的坐标系。后面所有章节都假设你已经有了这个坐标系。

我先带你过一遍 FDE 一天的样子，再把一个项目从头到尾的四个阶段讲清楚。

---

## 1.1 FDE 一天

下面这一天不是某个具体公司，是把多个 B 端 AI 项目里"工作日的实际形状"压缩成一份。如果你在做 FDE，你的某些日子会和它非常像。

```
周二，客户现场。

09:00  客户运维 Slack 群里有人 @ 你 — "昨晚 RAG 召回率从 0.82 掉到 0.61"
09:15  打开 trace 看昨晚最后 200 条失败请求,发现 78 条都是
       "近 3 天新数据"类问题 — 客户业务部上周加了一批新条目但没
       同步进知识库
09:40  写两行重新 ingest 的 SOP 丢回群里
10:00  客户业务 PM 站会:他们想再加 4 个新意图,下周老板要看
10:30  跟客户工程一起对一个 OAuth 报错(不是你写的,但你最熟悉
       这个客户的网络拓扑)
11:00  和你们公司远端的资深同事拉个 30 分钟,讨论是不是该把
       检索拆成集成版,回去再做
12:00  午饭,客户食堂,听对面运营吐槽"上次那个仪表盘慢"
       — 你顺手记到本子上,未必这个项目用得到,但是个信号
13:30  写下午要 demo 的 4 个 prompt 变体,提前在评估集上跑一遍
       — 不能在客户面前现挂
15:00  Demo。客户业务负责人当场提了 2 个新场景,你不正面接
       —— "我回去拉一下 eval,明天给您过线评估"
17:00  把这 4 个新场景拆成 5 个边角样本 = 20 条新评估
17:30  灰度发布昨晚那个 ingest 修复,先 10% 流量
18:00  写日报:今天的进度 / 明天 3 件事 / 1 个风险
```

如果你数一下，这一天里"敲 IDE 打代码"的时间大概 25%。但**你做的每一件事都是工程动作**——分诊召回率下降是工程，把 4 个新场景翻译成评估样本是工程，灰度发布是工程，写 SOP 也是工程。

FDE 区别于"驻场实施工程师"或"售前架构师"的，正是这种**工程动作的密度**：表面上你在沟通、对接、演示，但每一步背后都是一个具体的工程判断在驱动。

理解这一点之后再看时间分配就不会困惑了——FDE 一天的"代码时间不到一半"不是因为不专业，是因为剩下的时间里你也在做工程，只是工程的形式变了。

## 1.2 一个项目的四个阶段

把镜头从"一天"拉远到"一个项目"。一个典型的 FDE 项目大约 12 周，会走过四个阶段。这四个阶段不是 FDE 圈内行话，是从大量项目复盘里抽出来的共性结构，对应客户那边在每个时间段最关心的是什么。

```
        Week 1-3            Week 4-7           Week 8-10          Week 11-12
        ─────────           ─────────          ──────────         ─────────
        Discovery           Scaffolding        Production         Handoff
        发现真问题            搭最小闭环           上线 + 稳定         交给客户
```

**Discovery（发现）** 是项目最值得多花时间的阶段。这个阶段你在干一件事——把客户嘴里那个"我们想做一个 AI 助手"，翻译成"具体哪个角色在哪个工作流里要解决哪个问题，成功长什么样、用什么数字衡量"。

为什么这个阶段最贵？因为绝大多数项目在这里就走偏了，但走偏的成本要到 Scaffolding 甚至 Production 阶段才显现出来。客户的 PM 说"我们要做 agent 自动写邮件"，你听了就开始搞工具集成、写编排逻辑——三周后做完发现客户真正紧迫的是"销售月底搬数据进 ERP"，是个一上午就能写完的小自动化。这种事在 FDE 圈子里反复发生，根因永远是 Discovery 没做透。

**Scaffolding（脚手架）** 是把 Discovery 的结论变成最小可工作版本。这里"最小"很关键——不是 demo，是能跑评估集出分数、能让一两个客户业务方真的用一周的小系统。这个阶段最容易犯的错是**把 demo 当 Scaffolding**。Demo 给老板看一次就过了，Scaffolding 是给真实用户用一周以上、能持续收到反馈的——这两件事的工程要求差很远。Scaffolding 阶段最重要的产物不是代码，是**评估集 v0 + 一份能跑分的脚本**——这两样有了，后面所有的优化才有方向。

**Production（产品化）** 是把"两三个用户用一周"扩到"几十个用户用几个月"。这个阶段大头不在功能，在**抗操作变形和故障**。客户那边的网络会抖、上游数据会脏、有人会用 Excel 复制一段 1 万字的工单粘贴进来。Production 阶段你写的代码 30% 是处理这些边角情况，30% 是监控告警，剩下的才是新功能。这个阶段第一次让 FDE 觉得"工作变成运维"——不是错觉，是真的变了。

**Handoff（交接）** 是项目最容易被低估的阶段。客户能用 ≠ 客户能维护。Handoff 做不好，半年后这套系统在客户那边坏掉、没人会修、最后被弃用——你这一整季的工作就白做了。这个阶段的产物是 runbook、培训材料、客户内部接手人的代码权限和上线流程。如果 Handoff 阶段你还在写新功能，几乎可以确定这个项目走不长。

### 这四个阶段的边界很模糊

Discovery 永远不"结束"——你做到 Production 还会发现新的客户痛点；评估集永远不"够"——上线半年还在补样本。但**每个阶段有一个主任务，你 70% 的时间应该花在主任务上**。

Discovery 阶段你 70% 时间在跟人聊、看客户怎么真实工作，不是在写代码。Scaffolding 阶段 70% 时间在跑评估、调 prompt、改 retrieval，不是在做客户演示。Production 阶段 70% 时间在看 trace、修边角 bug、加监控。Handoff 阶段 70% 时间在写文档、培训客户工程师、做权限和发布流程的转移。

如果某一周你 70% 时间和当前阶段的主任务对不上，那是个信号——要么你**搞错了自己在哪个阶段**，要么**你逃避当前阶段最难的事**。

### 怎么判断自己在哪个阶段

90% 的 FDE 项目失败案例可以归结成"搞错了阶段"。最常见的是以为自己在 Scaffolding，其实还在 Discovery，结果做了一堆没人要的功能。下面这四个问题按顺序问，第一个回答"否"的就是你当前所在的阶段：

1. **客户的"成功"具体到一个数字了吗？** 不是"我们想要 AI 助手帮提效"，是"工单分诊准确率达到 90%、平均处理时间缩短 30%"。如果回答否，你在 Discovery，不应该写代码。

2. **你有评估集，能跑出当前版本的分数吗？** 不是"客户觉得不错"，是有一份固定的样本集 + 一份脚本，能 30 分钟内告诉你 v0.3 比 v0.2 强还是弱。如果回答否，你在 Scaffolding，先建评估集，再做演示。

3. **有客户的人在真实生产环境里用了至少一周吗？** 不是"试过"，是工作流里离不开它了。如果回答否，你在 Production，重点是稳定性和监控。

4. **客户内部已经有人能不依赖你独立运营吗？** 包括上线、修边角 bug、加新意图、看监控。如果回答否，你在 Handoff，写 runbook 和做培训。

四个都"是"——项目结束，做经验提取（这部分见第 16 章）。

## 1.3 LLM 应用 FDE 和数据交付 FDE 的差异

同样的一周，不同公司的 FDE 干的具体活儿可能差很远。最常见的两种形态：

**LLM 应用 FDE**——做 AI 助手、Agent、RAG 应用的公司。一周里大头是看 trace、调 prompt、跑评估、改检索、对接 API 限流和缓存。技术栈以 LLM 调用 + RAG + agent 为主。Demo 的形式是"场景化对话 + 准确率指标"。

**数据交付 FDE**——做企业数据平台、数据中台、数据本体（ontology）的公司，Palantir 是这类公司的代表。一周里大头是写 ETL、建数据模型、跑数据校验、对接客户的数据库连接和网络白名单。技术栈以数据管道 + 数据建模 + 仪表盘为主。Demo 的形式是"仪表盘 + 数据正确性"。

这两种形态在 FDE 这个职业上是**同一个心法、不同的肌肉**。心法都是"把外部能力装到客户身上、对结果负责而不是对代码负责"。肌肉是技术栈和工具链不同。

更重要的是，**同一家公司同一个项目的不同阶段也会切换**。Discovery 阶段你大概率是数据交付的姿势——读客户的 schema、画他们的工作流、看他们用什么报表。Scaffolding 之后才切到 LLM 应用的姿势——调 prompt、跑评估、看 token 成本。Handoff 阶段又切回去——你交给客户的不只是模型 endpoint，是一整套数据接入和监控。

理解这件事的好处是：当你下一个项目里发现自己在做"上一份工作没碰过的事"，不需要焦虑。换姿势不换心法，几周内你就能上手。

> **关于"FDE"这个名字**：FDE 作为一个统一称呼并不是行业里所有公司都采用的官方头衔。Palantir 把它当成核心职位名；做 AI 落地的工程师在不同公司里叫法不同——Solutions Architect、Customer Engineer、Delivery Consultant、Deep Learning Architect 都见过。本书统一用"FDE"是因为它最贴切地概括了这个角色"驻在客户那边、对结果负责"的本质。你在简历上见到的可能是另外的字，但只要工作内容是 1.1 节那种节奏，本书的内容就适用。

## 1.4 时间花在哪儿是对的

这一章最后一节给三个简单的自检问题。每两周问自己一次，比看任何项目管理工具都管用。

**第一，你这两周有几个小时花在"非编码工程"上？** Discovery 访谈、评估集标注、读客户业务文档、写 runbook、看 trace 找模式——这些都算非编码工程。如果两周加起来不到 16 小时，你大概率把项目做成了"远程外包"。

**第二，你这两周主动去客户工位边或食堂的次数有几次？** 不是开会，是你自己凑过去的。看一线员工怎么真用你的产品；和运营、客服、销售吃顿饭；周五下班前蹲在工位看他们关闭电脑前的最后两个动作。如果两周一次都没有，你的"对客户的理解"很可能正在变虚——你看到的是客户领导让你看到的东西，不是真实工作流。

**第三，你这个月有没有主动反对过客户的某个需求或方案？** 不是抬杠，是有数有据地拒绝。"客户要做全公司多 agent 协同——我看了他们的数据接入情况，建议先做单 agent 加工具调用。"或者"客户想私有部署 70B 模型——我做了推理成本测算，托管 API 在他们的 QPS 下便宜 5 倍。"如果一个月一次反对都没有，你已经退化成了"高级实施工程师"——客户说什么你做什么，没在用判断力。

这三个问题一个月做不到，你需要主动调整工作节奏。三个问题三个月做不到，你应该认真考虑这份工作还合不合适。
