# Part II: Customer Discovery — Requirements, Engineer-Style

> Scope: applies to both tracks.

---

## What This Part Solves

The most expensive code on an FDE project is the code nobody wanted.

And "code nobody wanted" almost always traces back to one cause: **Discovery wasn't done properly**.

Plenty of engineers hear "Discovery" and think "that's the PM's job." Wrong. FDE Discovery is not the PM flavor — sit in a meeting, listen to requirements, write a PRD. It is **observing, quantifying, and prototyping** the customer's actual workflow with an engineer's eye, and converging on an executable eval set and a SOW.

**Skip Discovery → the faster you ship engineering, the further off-course you go.**

This Part teaches two things:

1. How to run one round of Discovery using three moves: observe + ask + prototype.
2. How to translate Discovery findings into eval set + acceptance criteria + SOW.

## Chapters

- **Chapter 4: Discovery — observe before you ask.** How to read a customer's workflow; the 12 questions you must always ask; three field-observation techniques; a Discovery report template.
- **Chapter 5: From requirements to acceptance — eval set / acceptance criteria / SOW.** How to translate Discovery into engineerable contractual artifacts; what an eval set v0.1 looks like; a SOW template.

## How This Part Connects

- **Prerequisite**: the coordinate system from Part I's three chapters.
- **Downstream**: this Part's outputs (eval set + SOW) feed Parts III, IV, V, and VI in their entirety. If Discovery is weak, every line of engineering after it is wasted.
