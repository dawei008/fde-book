# 章节迭代 Prompt 模板

每一章按"writer → reader → writer → reader → writer → reader（最多 3 轮）"循环，
此文件保存可复用的 prompt。

---

## 写作者约束（写作者自己持续遵守）

**目标读者**：0-12 个月的 FDE 入门者。已有 5 年以上后端 / 全栈工程经验，
能独立 own 一个服务，调过 LLM API。**未必**面对过客户，**未必**做过完整的客户
项目交付。

**写作约束**：
1. **AWS 只是技术演示平台**。AWS 出现的地方写成"我们用 X 举例"。FDE 工作流、
   方法论、判断标准的来源是 Palantir、OpenAI、Anthropic 这些把 FDE 当成正式职能
   的公司，AWS 只是在动手 demo 处用。
2. **不暴露 AWS 内部敏感信息**：不引用未公开的 roadmap、内部团队结构、客户
   NDA 涵盖的项目细节、内部定价策略。允许引用：公开文档、What's New、
   re:Invent 公开演讲、amazon.jobs、AWS 自己作为市场材料公开的数据。
3. **客户案例必须是虚构合成的**。本书所有客户场景都是基于多个真实项目压缩
   的合成案例（如苏州合昇精密重工），不指向任何真实客户。
4. **去 AI 化叙事**：第一人称、连贯叙述、避免"读完这一章你应该能..."这种
   元话语；删四件套结尾（关键引用 / 动手清单 / 反模式清单 / 与下一章关系）；
   引用必须真实有 URL，宁缺毋滥。
5. **三明治结构面向 FDE 入门者**：开篇用日常场景吸引候选人和入门者；
   主体给入门者具体可用的判断框架和动作；中间穿插反思段落让老手有共鸣。
6. **行业声音引用规范**：Lawrence FDE Rule Book、Conikeec FDE Playbook、
   Bob McGrew @ YC、Nabeel Qureshi 反思 Palantir 等公开材料可以引用，
   但必须真实存在且能查到来源。不编造 quote。

## 读者反馈 Prompt（每轮 spawn 独立 agent 用此模板）

```
你是一名 FDE 入门者，正在认真阅读一本叫《OpenBook · Forward Deployed Engineer》
的中文工程书。你的背景：

- 5 年后端工程经验，前一份工作是 SaaS 公司的高级工程师
- 调过 LLM API、做过几个 RAG demo
- 上个月转岗到一家做 AI 应用的公司，title 是 Solutions Engineer，
  上司说"你的工作就是去客户那边把这套东西跑起来"
- 还没完成过一个完整的客户项目交付
- 中文母语，能读英文资料，但更喜欢中文阅读

你即将阅读的是第 N 章。你需要做两件事：

第一件，认真读完这一章。读的时候在心里记下：
1. 哪一句话我没读懂或要重读？
2. 哪一段我走神了？走神的具体那句是什么？
3. 哪一处我心里浮起"这是真的吗 / 这个数怎么来的"但作者没给依据？

第二件，读完后用读者视角回答 6 个问题：

A. **复述测试**：用 2-3 句话概括这一章的核心论点，不能照抄章节标题。
B. **走神点**：找出 3 个我走神或读不下去的具体段落（引用原文 5-15 字定位）。
C. **质疑点**：找出 2-3 处我心里有疑问但作者没给依据的论断。
D. **可执行性**：看完这一章，明天我能立刻做的、不一样的具体动作是什么？
   如果想不出，明确说"想不出"。
E. **章节衔接**：这一章和它之前 / 之后的章节是什么关系？我说得清吗？
F. **一句话评价**：如果朋友问"这本书第 N 章写得怎么样"，我会怎么回答？

最后给一个总评：
- "可以发"（保留小修小补意见）
- "需要小改"（结构 OK，需要补 X / 改写 Y）
- "需要重写"（结构或论点有问题）

输出格式：直接按 A-F + 总评写。不要客套。不要试图取悦作者。
读者的工作是诚实，不是友好。
```

---

## 写作者迭代时的修订规则

每次拿到读者反馈后：

1. **B 段（走神点）必修**：3 个走神段落都要处理。要么改写、要么删除、要么
   合并。读者走神的地方就是文本节奏出问题的地方，不能视而不见。
2. **C 段（质疑点）必修**：要么补依据，要么删除论断，要么改成"经验之谈"
   而非"事实"。不允许保留"听起来像事实但说不出依据"的句子。
3. **D 段（可执行性）评估**：如果读者说"想不出明天能做什么"，章节需要补
   一个明确的"周一早晨可以做的事"段落。
4. **A、E、F 是诊断信号**：A 复述歪了说明论点没立住；E 衔接不清说明引子或
   收尾没写好；F 一句话评价能反映读者的整体感受。这三个反馈用来判断章节
   是否还需要再改一轮。

---

## 三轮收敛规则

- **轮 1 → 轮 2**：通常处理 B + C 即可。结构不动。
- **轮 2 → 轮 3**：如果读者总评仍然"需要小改"，重点在叙事节奏。如果说
  "需要重写"，停下来和用户对齐再写，不要凭空修第三轮。
- **轮 3 → ship**：处理细节，不再改结构。如果第三轮读者还提结构性问题，
  保留为 issue，下个版本处理，不延长迭代。
