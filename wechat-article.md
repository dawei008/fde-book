## 一、一个被严重低估的角色

2026 年了，AI 行业最火的几个岗位你都听过：算法、Prompt 工程师、AI 产品经理、ML 平台、Agent 工程师。

但还有一个岗，正在悄悄成为 OpenAI / Anthropic / AWS / Palantir 共同**抢人**的对象，却很少被中文圈讨论：

**FDE — Forward Deployed Engineer，前线部署工程师。**

最早是 Palantir 把这个岗位带到聚光灯下：派工程师驻场到客户现场，把 AI 真正变成业务结果。后来 Y Combinator 在 *Field Engineer Sprint* 里反复推荐这个模式，OpenAI / Anthropic 把这个岗位"高薪化"，AWS 的 GenAI Innovation Center 把它工程化、流程化。

但在大多数中文 ToB 团队里，这个岗位长这样：

- 名字叫"售前 + 解决方案 + 实施 + 客户成功"四合一
- 头衔写"高级方案工程师"或"AI 落地专家"
- 老板期待他既能写 RAG 又能跟 CIO 谈合规
- 但**真正能做这件事的人**，整个公司可能只有 2-3 个

我花了三个月时间，把自己过去几年和身边一线 FDE 的经验沉淀成了一本中文书：

> **OpenBook: Forward Deployed Engineer**
> **— AI 应用的落地工程学**
>
> `Outcome = Harness × Customer`
>
> 7 个 Part / 17 章 / 4 附录, 全部基于 AWS Bedrock 生态实操

这篇文章是这本书的**精华提炼**。如果你正在做 LLM 落地，或者你的团队里有人在做"客户驻场"工作，希望你能从中拿到至少 3 件可以下周就用的事情。

## 二、FDE 工程师 = 工程师里"最 PM"，PM 里"最工程师"

```
        FDE 不是
      ──────────────────────────────
       ❌ 售前(只画图不写代码)
       ❌ 实施(只装机不懂业务)
       ❌ 算法(只调参不见客户)
       ❌ 客户经理(只签单不交付)

        FDE 是
      ──────────────────────────────
       ✓ 见客户 + 写代码 + 看 dashboard + 算成本
       ✓ 跟业务老师傅吃饭 + 跟安全团队对线
       ✓ 一个人就能把"需求 → 验收"端到端走完
```

简单一句话：

> **FDE 的工作不是写代码，是把客户业务变好；写代码只是手段之一。**

但这恰恰是它的稀缺性所在 —— 大多数人擅长其中一件事。两件事都擅长的人，再加上"愿意坐 8 个小时高铁去客户现场"这件事，就成了真正稀缺的 FDE。

## 三、三条铁律 ——FDE 一切判断的根基

我在书里第 2 章把这三条总结成"FDE 的三条铁律"。这三条决定了 FDE 跟普通 AI 工程师的判断方式根本不同：

### 铁律 1：Sell Outcomes — 不卖功能，卖结果

普通工程师说：
> "我们做了 RAG 系统，准确率 85%。"

FDE 说：
> "客户的法务以前看一份合同 2 小时，现在 25 分钟。一年节省 8000 小时。"

**翻译过来就是**：客户付钱不是为了 RAG，是为了"少招 5 个法务"。如果你不能把技术结果翻译成业务结果，所有 demo 都只是 demo。

### 铁律 2：Fix Forward — 不回头，往前修

```
  传统模式: 出问题 → 回滚 → 复盘 → 重做
  FDE 模式: 出问题 → 灰度上修 → 同步打补丁 → 不停服务
```

为什么？因为客户业务**等不起回滚**。一个 12 周的项目，如果上线第二周就回滚到上线前，你前 12 周都白干了。FDE 做的所有架构决策，都默认"未来要边跑边修"。

### 铁律 3：Eval-Driven — 没 Eval，不写代码

这条最反直觉，但也是最重要的。99% 的 LLM 项目失败，不是模型不行，是**没人知道什么叫"做好了"**。

FDE 第一周做的不是搭脚手架，是**逼客户业务专家手写 200 道带标准答案的题**。这 200 道题就是项目的"宪法"。后面所有的开发、上线、验收，都按这本宪法跑。

> "*Without an Eval set, you're not engineering — you're praying.*"
> — 这是我抄进书里的一句话，献给所有 0 Eval 就开干的团队。

## 四、Discovery 周 — 一个被严重忽视的 7 天

中国 ToB 项目失败的最大单一原因：**Discovery 没做好就开干**。

FDE 在 Discovery 周做的事，不是问问题，是**观察**：

```
  普通顾问问:  "你的痛点是什么？"
  FDE 做的:    去现场坐 1 天, 看业务真正怎么干
```

我书里第 4 章给了一份 **15 分钟初次访谈大纲**，覆盖业务、数据、技术、合规、决策五大块。这份大纲我自己用过几十次，每次都能在第一次会议里挖出"客户没说但是要命"的信息。

举个例子：

```
  客户说:    "我们想做 AI 自动核保。"

  你听到:    "RAG + Agent，简单。"

  Discovery 真相 (现场观察 1 天后):
    - 80% 的核保是标准件, 现行规则引擎已经在做
    - 另外 20% 是"个案", 需要 5 个老师傅经验决策
    - 客户真正的痛是 "5 个老师傅都要退休了"
    - 这是一个 知识保鲜 + 决策建议 项目, 不是自动化
    - 用 Agent 全自动 = 两年不可能上线
    - 用 RAG 给老师傅做 "决策建议" = 6 周可上线
```

**这就是 Discovery 的价值：在还没写一行代码前，把项目从"两年失败"变成"6 周成功"。**

## 五、技术栈速决 —— 30 分钟做完选型

LLM 这两年技术栈爆炸：30 个模型、40 个框架、20 个向量库、15 个 Eval 工具。FDE 不能花一个月选型，因为客户给的窗口就是 6 周。

我在书里第 6 章给了一张**速决矩阵**，每个分类下 3-5 个候选 + 关键选型维度。客户接进来 30 分钟内，就能定下"模型 + 框架 + 向量库 + 部署 + Eval"5 块默认配置。

举一个**最常见的客户场景**：

```
  场景: AWS 中等保险公司 + 文档 RAG + VPC 内
  ───────────────────────────────────────────
  模型:   Bedrock Claude 3.5 Sonnet
  框架:   Bedrock Knowledge Bases + Lambda
  向量:   OpenSearch Serverless
  Eval:   LangFuse Cloud + Promptfoo CI
  部署:   Lambda + API Gateway
  IaC:    AWS CDK
  监控:   CloudWatch + X-Ray
  安全:   IAM Identity Center + Bedrock Guardrails
```

**为什么这套？因为客户已经在 AWS 生态里。**

FDE 的选型逻辑很简单：

> **客户已经有的，优先用。客户没有的，选最像客户已有的。**

这不是技术上"最优"，但绝对是项目上"最快"。FDE 的任务是 6 周让事情跑起来，不是让客户多买一套数据中台。

## 六、决策树 ——RAG / Fine-tune / Prompting / Agent

这是 LLM 工程师永远在争的问题。我在书里第 7 章给了一棵决策树，浓缩成一张图：

```
        客户痛点
        ───────────────────────────

        客户说"我们文档太多, 找不到答案"
            ↓
        → RAG (90% 答案在这里)

        客户说"AI 回答不像我们公司风格"
            ↓
        → Prompting + 几条 fewshot (改 prompt 就够)
        → 真有大量风格语料 + 预算 → fine-tune

        客户说"AI 答得对但不够细致 / 不够专"
            ↓
        → 优化 RAG (chunking + rerank)
        → 加领域 fewshot

        客户说"我希望 AI 自己干活, 不只是回答"
            ↓
        → Agent (但要先 Eval 评估代价)

        客户说"我们要让 AI 操作我们内部 SaaS"
            ↓
        → Agent + MCP (Model Context Protocol)
```

**FDE 默认偏好顺序**：

```
  1. Prompting 能搞定 → 不上 RAG
  2. RAG 能搞定 → 不上 Fine-tune
  3. RAG + 工具调用能搞定 → 不上 Agent
  4. Agent 能搞定 → 不上多 Agent 框架
```

为什么？因为越复杂越难维护，FDE 撤离后客户接不动。**简单的方案 = 客户能接住的方案**。

## 七、PoC → 生产 — 90% 项目死在这里

业内有一个数字：85% 的 LLM PoC 上不了生产。我自己的观察 —— 这个数字对，而且原因高度一致：

```
        PoC 跟生产差 100 倍
        ─────────────────────────────────────────

  指标             PoC          生产
  ─────────       ──────       ───────────
  并发              1-5         100-1000
  数据量            10 文档     100K-1M 文档
  延迟              不在乎      P95 ≤ 3s
  可用性            尽力        99.5%
  成本              不在乎      预算可控
  审计              没          全链路 trace
  权限              admin       per-user RBAC
  回滚              改代码      5 分钟自动
```

把 PoC 推到生产 = 把这 8 个维度全部从"PoC 级"提升到"生产级"。

我在书里第 12 章给了一个"PoC 必做的 6 件事"清单：

```
  1. 写 Eval 集 (200 条 + CI 跑)
  2. 监控 dashboard (6 张卡片: 准确率/延迟/成本/QPS/失败/会话长度)
  3. 灰度部署 (canary 1%→100%)
  4. 5 分钟回滚 SOP
  5. 成本控制 (caching + routing + 预算告警)
  6. 全链路 trace_id (出问题能查到根因)
```

这 6 件事**做不完不上线**。FDE 在 W6 就要把这 6 件事铺好脚手架，不是 W11 才想。

## 八、灰度 + 回滚 + 成本 —— 三件第一周就要做的事

新人 FDE 最常犯的错：把灰度、回滚、成本控制留到上线前再做。结果上线一拖再拖。

**老手 FDE 的做法**：第 1 周就把这三件事的脚手架搭好，哪怕只跑 5 个 Eval case。

### 灰度

```
  W1 上线: 1% 流量 (内部员工)
  W2:      10% (小批客户)
  W3:      50%
  W4:      100% (全量)

  每次切流前看 4 个指标:
    - Eval 通过率没掉
    - 错误率没升
    - 延迟没退化
    - 成本没失控
```

### 回滚

```
  必须 5 分钟内能回滚到上一个稳定版本。
  实现: API Gateway stages + Lambda alias + 一键脚本
```

### 成本

```
  4 个工程动作:
    1. Prompt caching (重复 prompt 部分缓存)
    2. Model routing (简单问题走 Haiku, 复杂走 Sonnet)
    3. Compression (上下文裁剪)
    4. Batching (异步任务批处理)

  这 4 件事做完, 单次成本能从 $0.10 降到 $0.02。
  100 万次/天的客户, 一年省 80 万美金。
```

## 九、Agent 时代 —— 工具集少即是多

2025-2026 年，企业 ToB 项目几乎全部要求 Agent + MCP 适配。但 Agent 是 LLM 工程里最难的一块，因为它失败模式多到吓人：

```
  Agent 失败的 Top 5
  ───────────────────────────────────
  1. 工具集太多 → Agent 选错工具
  2. 没沙箱 → 一次错误调用搞坏生产
  3. 没幂等 → 重试搞出 5 张相同订单
  4. 没 HITL → 写操作没人审就执行了
  5. 没监控 → 出问题查不到链路
```

我书里第 14 章总结了一个"魔法数字"：

> **每个 Agent 的工具集 ≤ 30 个，最好 ≤ 15 个。超过这个数，准确率会指数下降。**

```
  10 个工具:  准确率 90%+
  20 个工具:  准确率 80%
  30 个工具:  准确率 60%
  50 个工具:  准确率 35% (玄学)
```

工具多了就拆 Agent，**别让一个 Agent 管 50 件事**。

第 15 章讲 MCP（Model Context Protocol）—— Anthropic 在 2024 年 11 月发布的协议，2026 年已经成为企业 Agent 集成的标准。它把"N 个 Agent × M 个工具"的集成噩梦，变成"N 个 Agent + M 个 MCP server"的标准接口。

## 十、Handoff 不是最后一周 —— 是上线前 3 周开始

中国 ToB 项目最常见的死法 #2 (前面是没 Eval)：**FDE 走了, 系统就垮了**。

原因不是代码烂，是**客户没有真正接走**。

我书里第 16 章给了一份"3 周倒计时"：

```
  T-3 周: 找客户运维 owner + 写 Runbook 大纲
  T-2 周: 4 小时培训 + 客户亲手操作每个动作
  T-1 周: 客户 owner 跟着 FDE 影子运营
  T 上线: 客户主导, FDE 后排
  T+4 周: FDE 完全撤离 (但 on-call 待命)
```

**关键 4 个能力**，全部具备 = 真 handoff：

```
  1. 客户能独立 deploy 一次 hot fix
  2. 客户能独立读 dashboard 找根因
  3. 客户能独立跑 Eval
  4. 客户能独立处理 Top 5 故障
```

少 1 个 = handoff 没完成 = 项目还没交付完。

## 十一、模式提取 —— 让下一个项目快 5 倍

FDE 老手 vs 新手最大的差距，不在第一个项目，**而在第二个项目**。

```
  新手 FDE:  第 2 个保险客户, 又 12 周做完
  老手 FDE:  第 2 个保险客户, 4 周做完, 第 3 个 2 周
```

差在哪？老手有"项目后资产库"：

```
  ▸ 代码模板:    "保险 RAG/Agent 起手包 v3.2"
  ▸ 文档模板:    Discovery 模板 / SOW 模板 / Runbook 模板
  ▸ 决策卡片:    "客户问 X → 答 Y" 速记
  ▸ 反模式集:    "上一次是这样翻车的, 不要再做"
```

每个项目结束 1 周内，FDE 必须做 5 件事：

```
  1. 写 1 页项目复盘 (做对了什么 / 做错了什么)
  2. Code review 自己整个项目 (哪些能复用)
  3. 抽 3 个"决策瞬间" → 写成卡片
  4. 抽 1-2 个"如果重来怎么做" → 项目模板 v X+1
  5. 跟同事 brown-bag 1 小时 → 讲坑而不是讲赢
```

> "*Pattern extraction is the difference between a consultant and an engineer.*"

## 十二、T 字成长 ——5 年成为不可替代

最后一章，是写给所有想长期做 FDE 的人。

```
        T 字成长结构图
        ─────────────────────────────────

  工程横向 (5 块都要懂):
    数据 / LLM应用 / 云 / 平台 / 软技能

  + 行业纵深 (选 1 个深入):
    保险 / 金融 / 制造 / 零售 / 医疗 ...

  = T 字 = 5 年后不可替代
```

为什么是 T 字？因为：

```
  纯横向 (一字):  会很多东西但每个都浅
                   → 5 年还是"项目工人"
  纯纵向 (I 字):  某行业很深但工程不行
                   → 容易被 AI 工具替代
  T 字:           工程横扫 + 行业深耕
                   → 5 年后客户主动找你
```

我书里给了一个 5 年路径：

```
  Year 1: 入门     — 跟项目, 选行业
  Year 2: 站稳     — 独立项目 + 工程深 1 块
  Year 3: 形成专长 — 公司内"行业 X 老法师"声誉
  Year 4: 不可替代 — 客户主动找你, 不再公司分配
  Year 5: 选下一步 — Principal / Lead / 产品 / 创业
```

**最重要的一句话送给所有 FDE**：

> **Sell the outcome. Fix forward. Eval first. Hand it off. Extract the pattern.**

## 十五、5 个你下周就能用的事

不管你看不看这本书, 这 5 件事拿去先用：

### 1. 把"卖功能"改成"卖结果"

不要再说"我们用了 RAG, 准确率 85%"。改成"客户法务每份合同从 2 小时降到 25 分钟, 一年省 8000 工时"。

### 2. 第一周让业务专家手写 50 道题

不是问"你想怎样", 是要"具体的题 + 标准答案"。这 50 道题就是你项目的合同。

### 3. 工具集控制在 15 个以内

如果你的 Agent 有 30+ 工具, 准确率正在指数下降。拆 Agent, 别堆工具。

### 4. 第一周搭好灰度 + 回滚脚手架

不是上线前再搭, 第一周搭。哪怕只跑 5 个 case, 框架要有。

### 5. 项目结束 1 周内写复盘

## 写在最后

FDE 这个角色, 在国内会**急速放大**。

1. 国内 LLM 应用市场已从"试试看"切到"严肃落地" (2025-2026 拐点)

4. 海外大厂 (Palantir / OpenAI / Anthropic / AWS) 都在抢 FDE, 信号已经给出
